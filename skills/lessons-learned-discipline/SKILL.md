After any debug session that identifies a root cause, finding the fix is only half the job. The other half is making sure the same class of bug cannot recur — and landing that prevention at the right layer. This skill is the decision flow every agent runs before closing a debug ticket.

## The layer cake

From most durable to most personal:

| Layer | Where it lives | What it prevents |
|---|---|---|
| **Tooling** | CI step, pre-commit hook, lint rule, type guard | Whole class of errors caught mechanically on every future commit |
| **Guide** | `bytetalent/docs/guide-*.md` or repo `docs/guide-*.md` | Explains the "why" — engineers make the right choice when they understand the model |
| **Skill** | `bytetalent/docs/skills/<area>/SKILL.md` | Copy-pasteable pattern an agent applies without reading the full guide |
| **CLAUDE.md** | Repo `CLAUDE.md` | Repo-specific deviations from universal standards; context that applies in this codebase only |
| **Memory** | `~/.claude/memory/feedback_*.md` | How a specific user works — preferences, rituals, communication style |

Layers are additive, not exclusive. A single root cause can warrant multiple layers: a CI check that catches the violation mechanically, plus a guide section that explains the underlying model for humans.

## Decision flow

```
After root cause is identified:

1. Can it be detected mechanically (CI, pre-commit, lint, type guard)?
      YES → ship the check AND a brief mention in the relevant skill (so agents know the check exists)
      NO  → continue to step 2

2. Is it universal — true across all stacks and projects?
      YES → update the skill in bytetalent/docs/skills/<area>/SKILL.md
            (if the bug class is complex enough to require explanation, also add a guide section)
      NO  → update the consuming repo's CLAUDE.md

3. Is it about how the user works or communicates?
      YES → write a personal memory entry (feedback_<topic>.md in agent's memory dir)
      NO  → none of the above layers is needed; root cause may have been one-off
```

Key: **always start at step 1**. Mechanical enforcement trumps documentation every time. If you can write a three-line check that catches the error before a PR merges, do that first, then write the explanation second.

## Worked examples

### Example 1 — prettier→husky hook regression (2026-05-01)

**Root cause:** A PR that added husky committed a `.husky/pre-commit` file generated by running `husky init` from a Unix shell. On Windows worktrees, the shebang (`#!/usr/bin/env sh`) in the file triggers an "Exec format error" because Windows Git cannot execute POSIX shell scripts directly.

**Layer analysis:**
- *Can it be detected mechanically?* Partially — a CI lint check could flag non-POSIX hooks, but the failure only surfaces on Windows, so a Unix CI wouldn't catch it.
- *Is it universal?* Yes — any project using husky in a cross-platform team can hit this.
- *Decision:* Skill update in `bytetalent/docs/skills/` — note that husky `pre-commit` must use `node --` syntax or `npx lint-staged` so it works on Windows + Mac + Linux worktrees. No pure tooling fix available.

### Example 2 — Drizzle journal `when` timestamps (2026-05-03)

**Root cause:** Worker agents generated migration entries 0040–0048 with `when` values in 2025 (predating the last-applied 0039 which was 2026). Drizzle's migrator uses `lastApplied.created_at < migration.when` to decide apply vs skip — with stale timestamps, all new entries were silently skipped.

**Layer analysis:**
- *Can it be detected mechanically?* Yes — a script that reads `drizzle/meta/_journal.json` and asserts strict monotonicity can catch this in CI before a PR merges.
- *Decision:* Ship the CI check (`scripts/check-journal-monotonic.mjs`, wired into `.github/workflows/ci.yml`). Also add a guide section to `bytetalent/docs/guide-db.md` explaining the `when` semantics and the silent-skip failure mode. Also add a pitfall note in `bytetalent/docs/skills/drizzle-migration-patterns/SKILL.md` pointing at the guide section.
- *Result:* All three layers — the check is the mechanical defense, the guide explains the model, the skill callout ensures agents see it without reading the full guide.

### Example 3 — worktree contamination (2026-04-28)

**Root cause:** A worker operating in an isolated worktree wrote output files (images, temp JSON) to the parent repo's main worktree instead of the worktree's own path. The contamination surfaced as unexpected staged files on `dev`.

**Layer analysis:**
- *Can it be detected mechanically?* Partially — you could write a git-status check, but it would be noisy. The real prevention is agent behavior.
- *Is it universal?* Yes — any agent that spawns into a worktree could make this mistake.
- *Decision:* Memory entry (`feedback_worktree_contamination.md`) plus a note in the orchestration guide. The memory is sufficient for the current team; if the pattern recurs across multiple agents, escalate to a skill.

## When to escalate from skill to guide

A skill callout is a pointer. When the underlying model is complex enough that an agent following the skill pointer would still make mistakes without understanding the "why", write a guide section. Signals that a guide section is warranted:

- The bug involves a non-obvious invariant in a third-party tool (e.g., Drizzle's apply-skip semantics)
- The correct behavior requires understanding a sequence of events, not just a rule
- Multiple agents have hit the same bug from different starting points

## When NOT to add a layer

- One-off misconfiguration that only happened because of a temporary environment state — if there's no realistic recurrence path, don't add layers.
- Personal preference ("I like shorter commit messages") — memory only, not a skill.
- Something already covered in an existing skill — add a callout sentence in the existing skill rather than creating a new one.
